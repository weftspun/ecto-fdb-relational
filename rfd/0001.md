# ADR 0001: talk gRPC directly to `fdb-relational-server`, not JDBC/a JVM/a Rust NIF

## Status

**Superseded by [ADR 0003](#adr-0003-replace-the-grpc-transport-outright-with-adr-0002s-embedded-transport).**
This decision was v0.1's default transport; v0.2 removed the gRPC transport entirely
in favor of the embedded transport ADR 0002 proposed as an opt-in second option. The
reasoning below is kept for the historical record (why gRPC was chosen in the first
place, and what was actually verified about `fdb-relational-server`/`JDBCService`), not
because gRPC is still spoken by this adapter.

## Context

An earlier, *unverified* sketch for this adapter proposed: `Ecto` ->
`Ecto.Adapters.SQL` -> a `DBConnection` implementation -> a Rustler NIF ->
an embedded JVM running FRL's published JDBC driver
(`org.foundationdb:fdb-relational-jdbc`), with DDL translated ad hoc
per-statement. That sketch was never verified against a real server and had
real, foreseeable problems:

* An embedded JVM inside a BEAM NIF is unusual and risky: JVM startup cost,
  GC pauses, and long-running/blocking JDBC calls do not play well with
  NIFs (even "dirty" NIFs are meant to be bounded; a JDBC call can block for
  an arbitrary amount of time) and can starve the BEAM scheduler.
* It requires shipping and managing a JVM + the JDBC driver jar as a runtime
  dependency of an Elixir library, which is a heavy, unusual ask for
  `hex.pm` consumers compared to every other real community Ecto adapter
  (`postgrex`, `myxql`, `ecto_sqlite3`), none of which embed another
  language runtime.
* FoundationDB's own docs mark the whole JDBC interface "experimental and
  not production-ready" -- riding on top of it doesn't buy much stability.

## What was actually verified this session (not guessed)

1. `fdb-relational-server` (Maven Central,
   `org.foundationdb:fdb-relational-server:4.3.6.0:all`, main class
   `com.apple.foundationdb.relational.server.RelationalServer`, started with
   `-g <grpcPort> -p <httpPort>` and the `FDB_CLUSTER_FILE` env var) was
   stood up against a real FoundationDB cluster and connected to
   successfully via the published JDBC driver, in an earlier part of this
   work.
2. On startup it advertises three gRPC services:
   `grpc.relational.jdbc.v1.JDBCService`, `grpc.health.v1.Health`, and
   `grpc.reflection.v1alpha.ServerReflection`.
3. The `.proto` files for `JDBCService` are public, versioned source in the
   `fdb-relational-grpc` module of `FoundationDB/fdb-record-layer`:
   `fdb-relational-grpc/src/main/proto/grpc/relational/jdbc/v1/{jdbc,column,result_set,continuation}.proto`.
   They are vendored verbatim (with their original Apache-2.0 license
   header intact) in this repo at
   `priv/protos/grpc/relational/jdbc/v1/`.
4. `JDBCService` exposes exactly what an Ecto SQL adapter needs, with SQL
   text as the actual "query language" (not a bespoke structured query
   API):

   ```protobuf
   service JDBCService {
     rpc getMetaData(DatabaseMetaDataRequest) returns (DatabaseMetaDataResponse) {}
     rpc execute(StatementRequest) returns (StatementResponse) {}          // SELECT
     rpc update(StatementRequest) returns (StatementResponse) {}          // INSERT/UPDATE/DELETE/DDL
     rpc insert(InsertRequest) returns (InsertResponse) {}                // Direct Access API, not used by this adapter
     rpc get(GetRequest) returns (GetResponse) {}                        // Direct Access API, not used by this adapter
     rpc scan(ScanRequest) returns (ScanResponse) {}                     // Direct Access API, not used by this adapter
     rpc handleAutoCommitOff(stream TransactionalRequest) returns (stream TransactionalResponse) {}
   }
   ```

   `StatementRequest` carries `sql` (full text), `database`, `schema`, and
   positional `parameters` -- exactly the shape `Ecto.Adapters.SQL.Connection`
   already expects a driver to produce.
5. `protoc` (installed via `scoop install protobuf`, v35.1) plus
   `protoc-gen-elixir` (from the `protobuf` hex package, v0.17.0) generate
   real, compiling Elixir modules from those vendored `.proto` files with
   no manual editing -- see `lib/ecto_fdb_relational/grpc/*.pb.ex`, which
   includes a real `Grpc.Relational.Jdbc.V1.JDBCService.Stub` gRPC client.

## Decision

Talk gRPC **directly** from Elixir to `fdb-relational-server`, using the
`grpc`/`protobuf` hex packages and the vendored, real `.proto` files. No
JDBC, no embedded JVM, no Rust NIF, no FFI of any kind.

```
Ecto.Repo
  -> EctoFdbRelational.Adapter            (use Ecto.Adapters.SQL)
  -> EctoFdbRelational.Adapter.Connection (Ecto.Adapters.SQL.Connection: SQL text + DDL builder)
  -> DBConnection pool
  -> EctoFdbRelational.Protocol           (DBConnection driver)
  -> GRPC.Stub / Grpc.Relational.Jdbc.V1.JDBCService.Stub
  -> fdb-relational-server (JDBCService, plaintext gRPC/HTTP2)
```

This is architecturally the same shape as every other real Ecto SQL
adapter (a `DBConnection` driver talking a wire protocol directly, wrapped
by `Ecto.Adapters.SQL`), just with gRPC/protobuf instead of a
Postgres/MySQL/SQLite wire format. It has none of the embedded-runtime
risk of the NIF+JVM sketch, and depends only on pure-Elixir/Erlang
libraries (`grpc`, `protobuf`, `db_connection`) already on hex.pm.

## Consequences

* No JDBC `DatabaseMetaData` introspection is available or needed --
  which sidesteps the `getTables()`/`getColumns()` bug/rough-edge noted
  during manual JDBC testing this session (`Database </frl/zftest6> does
  not exist` thrown immediately after successfully creating and populating
  that exact database). This adapter tracks schema state itself instead
  (see `EctoFdbRelational.SchemaTemplate` and `EctoFdbRelational.Ddl`) --
  which was going to be necessary regardless of transport, since FRL's
  schema-template DDL model doesn't match Ecto's incremental migration
  model either way (see the README's "Known gaps").
* `JDBCService`'s unary `execute`/`update` RPCs are effectively
  autocommit-per-call; real cross-statement transactions require the
  bidirectional-streaming `handleAutoCommitOff` RPC, which is **not**
  implemented in v0.1 (see `EctoFdbRelational.Protocol` moduledoc and the
  README). This is a real, load-bearing limitation of this decision, not
  hidden: it was true regardless of transport (JDBC's own
  `Connection.setAutoCommit(false)` presumably drives the same streaming
  RPC under the hood), but is called out here because it directly affects
  what `Repo.transaction/2`/`Ecto.Multi` can safely be used for today.
* The Direct Access API (`get`/`scan`/`insert` RPCs, which move data via a
  `KeySet` rather than SQL text) is not used by this adapter. It's a real,
  potentially more efficient path for point lookups, but adopting it would
  mean bypassing Ecto's SQL query builder entirely for those cases -- left
  as future work, not started here.

# ADR 0002: embed FRL in-process via a Rustler NIF + JNI, for one narrow goal

## Status

Proposed as additive to ADR 0001 (not a replacement) when written. **Implemented and
then made the *only* transport by [ADR 0003](#adr-0003-replace-the-grpc-transport-outright-with-adr-0002s-embedded-transport)**,
which also explains why "additive, opt-in" didn't hold up in practice. The design
below (what to embed, how the Phase 1 spike de-risked it) is otherwise still accurate.

## Context

ADR 0001 rejected `Ecto.Adapters.SQL -> DBConnection -> Rustler NIF ->
embedded JVM running FRL's JDBC driver` for concrete reasons: JVM
startup/GC cost and dirty-NIF blocking-time expectations clashing with
arbitrarily-long JDBC calls, the packaging burden of shipping a JVM +
jars with a hex.pm library, and FoundationDB's own docs marking the whole
JDBC interface "experimental."

None of that reasoning was wrong, and none of it is being revisited here.
What's changed is the *goal*: not "should this adapter talk JDBC instead
of gRPC" (still no), but "can this adapter avoid requiring a
separately-managed `fdb-relational-server` process at all," running
everything in one BEAM node instead. That is a real, narrower ask that
re-accepts the specific risks below in exchange for one specific benefit
(no separate process/port to start, monitor, or supervise).

This is **not** motivated by any current bug. While building this
adapter's CI integration tests (see the `integration` job in
`.github/workflows/ci.yml`), the bootstrap DDL sequence hit
`fdb-relational-server returned gRPC status 2: Database <path> does not
exist` for a database created moments earlier in the same setup. Tracing
FRL's own source (`fdb-relational-core`'s `EmbeddedRelationalDriver` /
`RecordLayerStorageCluster.loadDatabase`) confirmed this gRPC server
handler calls the *exact same* embedded-connection code an in-process
JDBC/embedded driver would call -- and this repo's own "Consequences"
section above (written earlier, independently) already documents hitting
the identical symptom via raw JDBC testing, before gRPC was ever
involved. **This is a bug in FRL itself, reproducible through either
transport.** Embedding FRL in-process would not fix it.

## What was verified this session (not guessed)

* `fdb-relational-jdbc` (FRL's published remote JDBC driver, `jdbc:relational://...`)
  is **not** what's needed here -- that module talks gRPC to a running
  `fdb-relational-server`, i.e. it's the Java-side equivalent of what this
  adapter already reimplements in Elixir. The actual in-process driver is
  `EmbeddedRelationalDriver` (`jdbc:embed:...`) in `fdb-relational-core`,
  which requires linking `fdb-relational-core` + `fdb-record-layer-core` +
  the native FDB client (`libfdb_c`) -- the same dependency weight
  `fdb-relational-server` itself carries, just loaded via JNI in-process
  instead of run as a subprocess.
* `org.foundationdb:fdb-relational-server:4.3.6.0:all` publishes a fat
  jar on Maven Central; `fdb-relational-core` does **not** (only
  javadoc/sources jars). There is no ready-made classpath for embedding
  -- assembling one (transitive dependency resolution, or a hand-built
  shaded jar) is real, non-trivial packaging work and the biggest open
  unknown in this decision.
* Rustler's dirty schedulers are still expected to complete in *bounded*
  time; an indefinitely-blocked JDBC call (e.g. an FDB network partition)
  is exactly the unbounded case they're not meant for. The safe pattern:
  a NIF call only enqueues work on a dedicated native OS thread and
  returns immediately; that thread performs the blocking JNI call and
  delivers the result asynchronously back to the calling process's
  mailbox (`rustler::env::OwnedEnv::send`).
* There is **no crash isolation**: a JVM segfault, native OOM, or a
  panic/exception crossing the Rust<->JNI boundary uncleanly takes down
  the whole BEAM node -- categorically worse than today's gRPC connection
  error, which is recoverable via ordinary supervision/reconnection. This
  is the central cost being accepted, explicitly, for "no separate
  process."
* Consumers of this transport need a compatible JDK installed and
  discoverable (`JAVA_HOME` / `libjvm.so` / `jvm.dll` on the loader path)
  plus the FRL jars on a classpath -- real new runtime prerequisites
  beyond "just Erlang/Elixir," unlike every other real Ecto adapter
  (`postgrex`, `myxql`, `ecto_sqlite3`) and unlike this adapter's own
  default gRPC transport.

## Decision

Add `EctoFdbRelational.Protocol.Embedded` as a second `DBConnection`
implementation, selected via `Ecto.Repo` config, alongside the existing
gRPC-based `EctoFdbRelational.Protocol`. The gRPC transport remains the
default; this one is opt-in for anyone who has decided the trade-offs
above are worth not running a separate process. Both transports share
`EctoFdbRelational.Query`, `EctoFdbRelational.Error`,
`EctoFdbRelational.Types`, and the DDL/catalog-routing logic worked out
while debugging the gRPC path (FRL requires catalog-level DDL --
`CREATE`/`DROP DATABASE`, `CREATE`/`DROP SCHEMA TEMPLATE` -- to run
against its well-known `/__SYS` system database, per its own
`SQL_Getting_Started` quick-start; this is an FRL requirement, not a
gRPC-specific one, so it applies identically here).

Implementation is staged specifically to de-risk the biggest unknown (the
classpath/packaging problem) before writing any Rust or Elixir code: a
pure-Java spike first proves an embedded connection + this adapter's
bootstrap DDL sequence works against a real FDB cluster and produces a
usable classpath/jar, *then* the Rustler NIF and `Protocol.Embedded` get
built on top of a proven foundation.

## Consequences

* Everything ADR 0001 said about JVM/GC/blocking-call risk and packaging
  burden is still true and is being accepted, deliberately, for this one
  transport -- not reversed, not forgotten.
* A crash in the embedded JVM is a crash of the whole node using this
  transport. Anyone choosing `Protocol.Embedded` over the default gRPC
  transport is choosing that trade explicitly; this must be documented as
  loudly as the README's "Known gaps" section documents everything else,
  not buried in a config option's moduledoc.
* The FRL bug described above (catalog existence checks failing right
  after the thing was created) is orthogonal to this decision and affects
  both transports equally. Fixing or working around it is tracked
  separately, not blocked on or solved by this ADR.

# ADR 0003: replace the gRPC transport outright with ADR 0002's embedded transport

## Status

Accepted, v0.2.

## Context

ADR 0002 proposed the embedded-JVM transport as an *additional*, opt-in option
alongside the default gRPC transport, specifically to avoid reversing ADR 0001's
reasoning. In practice, maintaining both was not worth it: the gRPC transport requires
a separately-managed `fdb-relational-server` process (a real operational burden for
every deployment and for this repo's own CI), and every risk ADR 0001 raised about
JDBC/an embedded JVM turned out to also apply, in practice, to running this adapter at
all -- there is no meaningfully lower-risk way to talk to FRL that doesn't also require
a JDK and FRL's jars somewhere in the stack. Given limited time to keep both transports
correct and tested, this adapter now embeds FRL in-process unconditionally and the
gRPC transport is removed, not merely deprioritized.

## What changed, concretely

* `EctoFdbRelational.Protocol` (the one and only `DBConnection` driver again, not
  `Protocol.Embedded` alongside a surviving default) now calls `EctoFdbRelational.Native`
  -- a Rustler NIF (`native/ecto_fdb_relational_nif`) that embeds a JVM via `jni`'s
  invocation API and calls `bridge.Bridge` (`native/frl_bridge`), which calls
  `com.apple.foundationdb.relational.server.FRL` directly. No `fdb-relational-server`
  process, no gRPC, no network hop for statement execution.
* The wire *shape* did not change: `bridge.Bridge.execute` takes and returns the exact
  same `grpc.relational.jdbc.v1.{StatementRequest,StatementResponse}` protobuf bytes the
  old gRPC transport sent/received (FRL's own `FRL.execute`/`FRL.Response` already speak
  these messages internally, gRPC or not -- see `native/frl_bridge`'s moduledoc). So
  `EctoFdbRelational.Types`/`Query`/DDL-catalog-routing logic needed **no changes**;
  only `Protocol`'s connect/execute/disconnect plumbing and `Error` changed.
  `lib/ecto_fdb_relational/grpc/*.pb.ex` moved to `lib/ecto_fdb_relational/proto/` with
  the `GRPC.Service`/`GRPC.Stub` modules cut out of `jdbc.pb.ex` -- everything else in
  those files is plain protobuf message codecs, still needed, still generated from the
  same vendored `priv/protos/`.
  the `{:grpc}` hex dependency is gone; `{:protobuf}` stays (message codecs only) and
  `{:rustler}` was added.
* `spike/jvm_embed` (ADR 0002's Phase 1 de-risking spike) is removed: it served its
  purpose (proving the classpath/packaging problem was solvable, and that the
  `UPDATE ... SET x = ? WHERE y = ?` planner bug reproduces identically in-process) and
  `native/frl_bridge` is now the real, non-throwaway version of the same idea. Its
  history is still in git if anyone needs to see the original pure-Java proof.
* Config changed from `hostname`/`port` (the gRPC address) to `cluster_file` (the
  FoundationDB cluster file path) -- see the README.

## Consequences (accepting ADR 0001's and ADR 0002's risks as permanent, not provisional)

* **No crash isolation, permanently.** A JVM segfault, native OOM, or a panic/exception
  crossing the Rust<->JNI boundary uncleanly now takes down every deployment of this
  adapter, not just an opt-in one. `EctoFdbRelational.Protocol.ping/2` is a deliberate
  no-op as a result -- see its moduledoc.
* **A JDK and a Rust toolchain are hard prerequisites to even `mix compile` this
  adapter now**, not just to run its integration tests. `EctoFdbRelational.Native`'s
  `use Rustler` links against `libjvm` (via `java-locator`/`JAVA_HOME`) as an ordinary
  part of compiling `lib/`, and `native/ecto_fdb_relational_nif/build.rs` compiles
  `native/frl_bridge`'s one Java class with `javac` plus the FRL jars on
  `ECTO_FDB_RELATIONAL_CLASSPATH` -- as part of `cargo build` itself, not a separate
  Mix compiler step (an earlier version of this ADR tried that; see build.rs's own
  comment for the real bug it caused: `use Rustler`'s `@on_load` starts the embedded
  JVM the moment `:elixir` compiles/loads `EctoFdbRelational.Native`, which is *before*
  a Mix compiler ordered after `:elixir` would have produced `priv/java` yet, and
  *after* one ordered before it -- which Mix can't even resolve, since a project's own
  custom compiler task isn't compiled yet either at that point). This is a real
  regression in "just add the hex dependency and go" simplicity versus every other real
  Ecto adapter (`postgrex`/`myxql`/`ecto_sqlite3`) and versus this adapter's own v0.1 --
  accepted because there was no time to keep a pure-Elixir transport working and correct
  alongside this one. See the README's install section for exactly what's required.
* **Every NIF call runs on Rustler's `DirtyIo` scheduler**, not the fully async
  "enqueue on a dedicated thread, deliver via `OwnedEnv::send`" design ADR 0002
  originally sketched -- a deliberate simplification to ship a working first cut, at
  the cost of tying up a dirty scheduler thread for the duration of each FDB call
  (including an arbitrarily long one, e.g. during a network partition). Revisiting this
  is plausible future work, not ruled out, if dirty-scheduler exhaustion becomes a real
  problem in practice.
* No per-call timeout is wired up (the old gRPC transport's `opts[:timeout]` ->
  `grpc_opts` plumbing had nothing to replace it with here -- a dirty NIF can't easily
  be cancelled mid-call). Documented as a known gap, not silently dropped.
* Everything ADR 0002 already said about JVM/GC/blocking-call risk being accepted "for
  this one transport" now reads as accepted for the *only* transport. Nothing in that
  reasoning was wrong; only the scope of who's exposed to it changed.
